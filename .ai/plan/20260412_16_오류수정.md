# 16차: RAG 복합 의도 라우팅 및 시스템 제어 토큰 표출 치명적 오류 수정 보고서

본 문서는 HR RAG 시스템에서 발생한 "검색 결과 무시(RAG Skip) 이슈" 및 "HITL UI 렌더링 상실(Token Sanitization) 이슈"의 근본 원인을 분석하고, 이를 수정하기 위해 시스템 파이프라인 및 백엔드 로직에 적용된 조치 범위를 기록한 기술 문서입니다.

## 1. RAG 검색 누락 (복합 의도 라우팅 버그)

### 1-1. 발생 현상
사용자가 "인턴 경험 있는 지원자 3명 합격 메일 보내줘"와 같은 **조건부 액션 요청(Search + Action)**을 입력할 경우, AI가 "인턴 경험이 있는 지원자 정보가 없습니다"라며 RAG 컨텍스트를 전혀 로드하지 못하고 작업을 중단하는 현상 발생.

### 1-2. 근본 원인 파악
라우팅 프롬프트(Routing Agent) 및 백엔드 의도 해석(`extractOptimizedQuery` 및 조건부 실행) 로직에서 단일 의도 구조(흑백 논리)의 설계 한계.
`RagService.java` 내의 RAG 벡터 검색 분기 코드:
```java
// 기존 코드
boolean isActionIntent = routingResponse != null && routingResponse.contains("INTENT: [ACTION]");

// ACTION 의도일 경우 뒤의 쿼리와 무관하게 무조건 검색을 스킵함
if (!isActionIntent && !optimizedQuery.equalsIgnoreCase("NONE")) {
    // RAG 검색 로직...
}
```
위 코드로 인해 분류기가 `ACTION` 의도로 판별하면 내부적으로 추출된 `QUERY:` 변수가 존재함에도 불구하고 RAG 모듈 호출이 **Short-circuiting(강제 종료)** 됨.

### 1-3. 해결 조치 사항
어떠한 메인 의도(Intent)를 갖더라도, 추출된 검색 쿼리(QUERY)가 유효하다면 즉시 RAG 검색을 우선 수행하도록 하이브리드 파이프라인으로 조건 완화.
- **분기문 수정**: RAG 검색 수행 조건을 `isActionIntent` 배제로 제한하던 논리 삭제.
- **분류기 프롬프트 강화**: "복합 명령(이 조건의 사람에게 작업을 수행해줘)이 들어오면 `ACTION` 의도라 하더라도 반드시 `QUERY:`에 대상 키워드를 적어라"라는 Few-shot 예시를 명시적으로 삽입.

```java
// 개선된 코드 (단일화)
if (!optimizedQuery.equalsIgnoreCase("NONE")) {
    // RAG 검색 무조건 수행!
}
```

## 2. HITL 메일 전송 승인 버튼/UI 누락 이슈

### 2-1. 발생 현상
일괄 메일 발송 툴(`sendBulkResultEmail`)이 백그라운드에서 정상 실행되었고 서버 로그에도 반환 토큰이 찍히지만, 정작 프론트엔드 채팅창 하단에는 승인/취소 처리를 할 수 있는 버튼 UI가 노출되지 않음.

### 2-2. 근본 원인 파악 (LLM Token Sanitization)
기존의 프롬프트가 LLM의 환각(Hallucination) 방지를 위해 제어 문자의 임의 창작을 강하게 억제한 것이 의도치 않은 **토큰 필터링 부작용** 야기.
- **이전 규칙**: *"3. `[APPROVAL:...]` 같은 토큰을 직접 작성하지 마세요. 5. 결과를 자연스러운 한국어로 요약해 전달하세요."*
- **LLM의 해석 오해**: 툴로부터 반환받은 원본 텍스트에 진짜 `[APPROVAL:BULK:...]` 토큰이 있었음에도 불구하고, LLM이 이것이 사용자에게 노출되면 안 되는 민감 제어 문자로 자의적 판단. 따라서 자연스러운 한국어로만 답변하고 **토큰 문자열을 고의로 삼켜버림(Sanitize).**
- **결과**: LLM의 최종 텍스트 출력 스트림에 해당 문자열이 애초에 존재하지 않았으므로, 백엔드의 `if (accumulated.contains(TOKEN_APPROVAL))` 파싱 인터셉터 로직이 발동하지 않아 프론트로 `approval` 채널의 SSE 이벤트가 미발송.

### 2-3. 해결 조치 사항
LLM이 자신이 만들어낸 것이 아닌 **'도구가 반환한 시스템 토큰 구조'** 만큼은 대답의 끝에 반드시 원본 그대로 100% 붙여서 출력하도록 프롬프트 정책(Policy) 강제 조정.

**개선된 프롬프트 제약 사항**
> "단, 당신이 도구(Tool)를 호출한 결과 메세지 안에 `[APPROVAL:...]` 또는 `[COMPLETED:...]` 토큰이 들어있다면, 그 토큰은 단 1글자도 요약/수정/생략하지 말고 당신의 최종 응답 텍스트 맨 마지막에 100% 원본 그대로 똑같이 포함시켜 반드시 출력해야 합니다."
> "(이 토큰이 누락되면 시스템 UI 버튼이 렌더링되지 않는 치명적 에러가 발생합니다.)"

## 3. 요약 및 향후 과제
- 이번 패치로 Action Tools와 RAG 간 논리적 파이프라인의 틈이 메워졌으며, 스트림 파싱 시 발생하는 LLM 텍스트 필터링의 취약성을 프롬프트 고도화를 통해 해결함.
- **Next Step**: 정규 표현식을 활용하여 백엔드에서 `[APPROVAL:...]` 구조체를 더욱 강건하게 파싱하거나, 향후 궁극적으로 네비게이션 기능과 동일하게 툴콜 시점에 직접 SSE 채널(`approval` / `completed`)을 쏘는 완전 토큰 분리 정책으로의 고도화를 고려할 필요성이 있음.
