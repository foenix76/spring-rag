# [백엔드] 라우팅 에이전트 도입을 통한 멀티 에이전트 최적화

단순 검색어 생성을 넘어, 사용자의 질문 의도를 분석하고 RAG 검색 실행 여부를 결정하는 '라우팅(Routing) 에이전트'를 도입하여 답변의 정확도와 성능을 동시에 잡습니다.

## User Review Required

> [!IMPORTANT]
> **라우팅 로직의 변화**
> 이제 첫 번째 에이전트가 "이 질문은 DB 조회가 필요하다/아니다"를 먼저 판단합니다. "도구 목록 보여줘" 같은 질문은 'RAG 생략'으로 판단되어 기존처럼 툴 기반 답변이 정확하게 나오게 됩니다.

## Proposed Changes

### [Component] Spring AI Backend Services

#### [MODIFY] [RagService.java](file:///home/foenix/workspace/spring-rag/src/main/java/com/example/korrag/service/RagService.java)
- **라우팅 에이전트(Router Agent) 구현**:
    - 질문 성격을 세 가지(`SEARCH`, `ACTION`, `GENERAL`)로 분류하도록 프롬프트를 재설계합니다.
    - 출력 형식을 규격화(예: `INTENT: [SEARCH] QUERY: [키워드]`)하여 Java에서 파싱하기 쉽게 만듭니다.
- **조건부 RAG 실행 로직**:
    - `INTENT`가 `SEARCH`일 때만 임베딩 및 벡터 검색(Rerank 포함)을 수행합니다.
    - 그 외의 경우 `context`를 비워두거나 시스템 메시지로 대체하여 불필요한 DB 조회를 방지합니다.
- **스트리밍 수집 및 결합**: 
    - 의도 파악 결과와 최종 답변 생성 과정을 매끄럽게 연결하여 답변 누락을 방지합니다.
