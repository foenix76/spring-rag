# 17차: 화면 이동 로그 중복 출력 및 토큰 파싱 에러 수정, LLM 컨텍스트 요약 보강

## 1. RAG 기반 특정 인원 추출 + ACTION 승인 / 취소시 버튼 재노출됨
특정조건 + 합격 메일 발송 시 메일 발송 처리 후 다시 메일 발송 버튼이 노출됨. 또한 reload 시에도 계속 노출됨.
(한번 발송 또는 취소시 처리상태가 확정이므로 버튼이 노출되지 않아야함)
또한 메일 발송 승인 요청 패널 우측의 고유번호도 N/A로 노출됨.
메일 발송 후 시스템메세지에 안내멘트 없이 체크박스 표시만 노출됨
버튼 재노출 오류 수정 및 채팅 저장시 처리 상태가 같이 저장이 되어야 함

## 2. 스트림 채팅 텍스트 중복 및 깨짐 (기형적인 Buffer 로직 제거)
**이슈:** 채팅창에 `[APPROVAL덕분에...` 형태의 깨진 문자열이 반복해서 나타나고 ID가 N/A로 바인딩되는 치명적 파싱 오류 발생.
**근본 원인:** 백엔드 `RagService`의 스트리밍 중간에 제어용 대괄호 `[` 가 한 번이라도 등장하면, 스트림 전송을 멈추고 `toolResultAccumulator`에 청크를 무한정 쌓아두다가 `]`를 만날 때 한 번에 방출하는 기형적인 필터 로직 때문. LLM이 `[참고 지원자 정보]에 의하면...` 와 같은 일반 문장을 생성하는 순간, 이 필터가 오동작하여 답변의 모든 텍스트를 흡수해버리고, 맨 마지막 토큰 발동 시 그동안 흡수한 모든 텍스트 덩어리를 통째로 중복 재전송하여 UI가 파괴됨.
**개선 조치:**
- 백엔드에 존재하던 "텍스트 지연 전송 및 필터링 로직"을 전면 삭제. LLM이 보내는 모든 텍스트 덩어리는 중간 필터링 없이 **무조건 즉시 프론트엔드로 전달**하도록 근본 분리.
- "백엔드는 단지 누적 텍스트에서 토큰이 등장했는지만 별개로 감지하여(Non-blocking) 토큰 이벤트만 이중 발송" 하도록 변경 (각 토큰당 스위치(AtomicBoolean) 적용).

## 3. 추가 강화 수정 내역 (문서강화 완료)
* **프론트엔드 Vue 상태관리 (Reactivity) 복원 및 Persistence 확보**:
  * 히스토리를 불러올 때(`loadHistory`) 컴파일 단계에서 객체 원형(`msgObj`)에 `isCompleted`, `isCancelled` 속성을 명확히 초기화하도록 수정했습니다.
  * `completedToken` 또는 `[시스템 취소]`, `[시스템 승인]` 등 사용자 발화 이력을 역추적하여, **원래 승인 패널(해당 태스크)** 의 본래 위치에 명확히 Link(상태 매핑)하는 2-Pass 스캔 방식을 도입해 재장전(Reload) 이후에도 버튼이 노출되는 문제를 영구 제거했습니다.
  * `TASK_ID`가 N/A로 노출되던 문제를 대처하기 위해 정규식 패턴 의존을 버리고, 포맷 구분자(`:`) 기준의 다중 조건 분기로 강력한 강제 파싱을 적용했습니다.

* **이동 (NAVIGATE) 채널 복구 및 호환성 방어**:
  * LLM이 슬래시를 빼먹거나, `interviews`냐 `schedule`이냐로 혼동되는 모든 사이드 이펙트에 대응할 수 있도록 `urlToView` 맵을 전방위(Universal Router Map)로 개편하여 면접 일정 등 모든 뷰로 완벽히 매핑시켰습니다.
  * Toast 메시지에 페이로드 객체가 직접 넘어와 `[object Object]`로 출력되던 이슈를 파싱 Fallback 로직 강화로 수정 완료했습니다.

해당 내용이 모두 GitHub 및 로컬 브랜치에 안전한 커밋으로 저장되었습니다! (UI 상태 동기화 및 NAVIGATE 보완)

## 4. LLM 할루시네이션(임의 포맷)으로 인한 실시간(Live) 승인 버튼 미종료 버그 수정내역
* **문제의 근본 원인**: `HrActionTools`의 실행 결과를 반환했을 때 LLM이 구조적 토큰을 가끔 빼먹거나, `[APPROVAL:sendBulkResultEmail,candidateIds=2026_034]` 처럼 함수 파라미터까지 지어내서 (Hallucination) 포맷을 파괴해버리는 현상을 브라우저 에이전트를 통해 포착했습니다.
* **이로 인한 사이드 이펙트**: 위 가짜 토큰을 기반으로 프론트가 버튼을 그렸는데, 막상 버튼을 누르면 실제 자바 백엔드는 `BULK_A8F2` 같이 정상적인 식별자를 생성해 `[COMPLETED:BULK_A8F2]`라며 다른 ID를 넘겨주므로, ID 불일치가 발생해 버튼이 가려지지 않는 원리였습니다. 
* **해결책 1**: 프론트엔드 실시간 SSE 파이프라인에서 Task ID 매칭 로직을 우회하여, **유저가 방금 버튼을 누른 메시지(processingTaskId 보유)** 를 추적 즉시 완료 처리하는 방식으로 개선했습니다.
* **해결책 2**: 자바 `RagService.java` 내의 메인 시스템 프롬프트에 `★★★ 구조적 토큰 취급 규칙 ★★★`을 추가해, 절대로 번역/할루시네이션 없이 100% 원본 유지 방출을 긴급 규칙으로 주입시켰습니다.
