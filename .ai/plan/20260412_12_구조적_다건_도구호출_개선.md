# RAG 하이브리드 라우팅 및 다건 도구 호출의 구조적 개선 (POC)

## 문제 인식
형의 예리한 지적대로, 프롬프트에 "절대 툴을 호출하지 말고 토큰을 직접 생성해라"라고 강제하는 방식은 근본적인 도구(Tool) 중심의 에이전트 구조에 어긋나. AI는 도구를 호출해서 그 결과를 기반으로 대화해야 하는데, 도구 호출을 막고 문자열을 찍어내게 하는 건 전형적인 Prompt Injection Hack일 뿐이지.

따라서 구조적으로 올바른 해결책은 **"다건 일괄 처리를 담당하는 전용 툴(Tool)을 제공하는 것"**이야.

## 주요 변경 사항

### 1. `src/main/java/com/example/korrag/service/HrActionTools.java` [MODIFY]
- **`sendBulkResultEmail` 툴 추가**:
  - 다건 발송을 위해 `List<String> candidateIds`를 받는 새로운 툴을 추가.
  - 이 툴 내부에서 `confirmed=false`일 때 `[APPROVAL_REQUIRED:ACTION:TARGET:ALL:TaskId]` 토큰을 반환.
  - 이렇게 하면 AI가 다수 대상자를 인지했을 때 자연스럽게 이 툴을 호출하고, 툴의 반환값을 사용자에게 텍스트로 전달하게 되며, UI가 이를 감지해 버튼을 렌더링함.

### 2. `src/main/java/com/example/korrag/service/RagService.java` [MODIFY]
- **프롬프트 정상화 (Hack 제거)**:
  - 기존의 강제성 짙은 프롬프트(`절대 먼저 툴을 호출하지 마세요...`) 제거.
  - 대신 "여러 명에게 발송할 경우 `sendBulkResultEmail` 툴을 사용하라"는 정석적인 가이드 제공.
  - 취소 시 툴을 재호출하지 말라는 기본 제약은 유지.

### 3. `src/main/resources/static/index.html` [MODIFY]
- **UI 승인 메시지 변경**:
  - `approveAction` 클릭 시 전송되는 메시지를 `sendResultEmail` 반복 호출 지시에서 `sendBulkResultEmail 툴을 confirmed=true로 호출하라`는 지시로 구조적 변경.
