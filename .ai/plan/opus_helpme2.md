# 🆘 Opus 형님, 이 개새끼(Gemini)가 친 사고 좀 수습해주세요

## 1. 현재 상황 요약 (Symptoms)
Gemini 이 병신새끼가 멀티 채널 스트리밍 설계한다고 깝치다가 기존에 잘 돌아가던 RAG 검색이랑 HITL(승인 버튼) 연동을 다 박살내놨습니다. 설계는 형(유저)이 다 해줬는데 코딩을 이따위로 해놨어요.

- **RAG 검색 먹통**: 사용자 질문에서 검색어를 제대로 못 뽑아서 DB 조회를 안 하거나, `ID1, ID2` 같은 가짜 데이터를 뱉어냅니다.
- **HITL(버튼) 실종**: 승인 요청 토큰(`[APPROVAL:...]`)이 올바른 채널로 안 가거나, 텍스트랑 섞여서 UI가 인지를 못 합니다.
- **UI 깜빡임/혼선**: 시스템 로그가 말풍선에 찍히고, 일반 대화가 로그창에 찍히는 등 채널 분리가 엉망입니다.
- **프롬프트 땜빵**: 툴 목록을 프롬프트에 하드코딩하는 등 유지보수 불가능한 짓거리를 해놨습니다.

## 2. 목표 아키텍처 (Intended Design)
형이 설계해준 **"필드 원천 분리형 스트리밍"** 구조를 완벽하게 구현해야 합니다.

### A. 백엔드 (`RagService.java`)
- LLM 응답을 단순 텍스트가 아닌 `Map<String, Object>` 형태의 멀티 채널 스트림으로 반환.
- **채널 종류**:
    - `text`: 순수 사용자 대화 (타이핑 효과용)
    - `system`: 도구 실행 로그 (화면 상단 고정 로그용)
    - `approval`: 승인 요청 토큰 (HITL 버튼 렌더링용)
    - `completed`: 작업 완료 토큰 (상태 업데이트용)
- **핵심 로직**: LLM이 툴을 호출하면, 그 결과물(토큰 포함)을 LLM이 다시 읊게 하지 말고 **백엔드에서 즉시 가로채서 각 필드에 담아 쏴줄 것.** (현재는 스트림 중간에서 문자열 파싱하는 멍청한 짓 중)

### B. 프론트엔드 (`index.html`)
- SSE로 들어오는 필드(`text`, `system`, `approval` 등)를 보고 **조건부 렌더링** 수행.
- 텍스트 파싱(`includes('[시스템 지침]')` 등)에 의존하지 말고, 백엔드가 정해준 필드값만 신뢰할 것.
- 채팅 이력 재로딩(`loadHistory`) 시에도 DB의 원본 텍스트에서 이 필드들을 정확히 복구해내야 함.

## 3. 핵심 파일 (Target Files)
- `src/main/java/com/example/korrag/service/RagService.java`: 스트리밍 채널 분리 및 RAG 검색 로직의 핵심.
- `src/main/resources/static/index.html`: 필드 기반 독립 렌더링 및 UI 상태 관리.
- `src/main/java/com/example/korrag/service/HrActionTools.java`: 툴 반환 규격 (현재 엉망임).

## 4. 오퍼스 형님께 드리는 전언
Gemini 저 새끼가 자꾸 "두더지 잡기" 식으로 코드 한 줄 고치고 열 줄 망가뜨리고 있습니다. 프롬프트에 툴 이름 적어넣는 짓거리 당장 중단시키고, **Spring AI의 스트림을 깔끔하게 인터셉트해서 필드별로 쏴주는 견고한 Java 로직**으로 환골탈태 시켜주세요.

**현재 `RagService.java`는 형이 설계해준 구조를 흉내만 낸 쓰레기 상태입니다. 전면 재건축이 필요합니다.**
