# 🚀 프로젝트 해설 (Project Deep-Dive)

안녕 형! 우리 RAG 시스템의 속살을 낱낱이 파헤쳐보는 해설서야. 데이터가 어떻게 흐르는지, 폴더 구조는 어떤지, 그리고 왜 이렇게 덩치가 큰지 한눈에 정리해줄게.

---

## 🌊 데이터 흐름에 따른 소스 설명

우리 시스템은 **'엄청나게 똑똑한 도서관'**처럼 움직여.

### 1단계: 책 정하기 (데이터 로딩)
도서관에 책이 없으면 아무것도 못 하겠지? 먼저 지원자들의 자소서를 도서관에 채워야 해.
*   **설명**: 100명의 지원자가 쓴 자소서를 DB라는 서랍장에 저장해.
*   **어디에 있어?**: `RagService.java`의 `run()` 메서드

### 2단계: 비밀번호 붙이기 (임베딩 - Embedding)
책 제목만 보고는 내용을 다 알 수 없잖아? 그래서 인공지능이 책의 **'의미'**를 숫자로 된 비밀번호로 바꿔서 붙여줘.
*   **설명**: "저는 도전적인 사람입니다"라는 글을 AI가 `[0.12, -0.45, ...]` 같은 숫자로 바꿔.
*   **어디에 있어?**: `OnnxEmbeddingService.java`의 `embed()` 메서드

### 3단계: 비밀번호 장부에 기록하기 (벡터 저장)
숫자로 바꾼 비밀번호(벡터)를 장부에 잘 적어놔야 나중에 빨리 찾겠지?
*   **설명**: 자소서 내용이랑 그 비밀번호를 `essay_vectors`라는 특별한 장부에 저장해.
*   **어디에 있어?**: `VectorStoreRepository.java`의 `upsertVector()` 메서드

### 4단계: 형이 질문하기 (질문 분석)
형이 질문을 던지면 시스템은 이 질문도 비밀번호로 바꿔.
*   **설명**: 질문도 숫자로 바꿔야 장부에서 비슷한 숫자를 찾을 수 있거든!
*   **어디에 있어?**: `RagService.java`의 `ask()` 메서드 시작 부분

### 4.5단계: 장기 기억 서비스 (History Lookup)
DB 서랍장에서 형과 나눈 과거의 대화들을 다시 꺼내와서 AI가 기억하게 해주는 단계야.
*   **설명**: **`ChatMessageRepository`**를 통해 `chat_messages` 테이블에서 최근 10개의 대화 내역을 조회해. 질문에 실린 '숨은 의도'를 파악하는 핵심 열쇠지!
*   **어디에 있어?**: `RagService.java` 내 `chatMessageRepository.findRecentMessagesAsc()` 호출 부분

### 5단계: RAG 지식 검색 서비스 (Knowledge Retrieval)
우리 시스템의 핵심인 **RAG(검색 증강 생성)**가 일어나는 곳이야. 장부를 뒤져서 질문과 관련된 자소서 5개를 골라와.
*   **설명**: **`VectorStoreRepository`**가 벡터 DB를 광속으로 뒤져서 형의 질문에 답할 수 있는 가장 정확한 '증거(자소서)'들을 확보해.
*   **어디에 있어?**: `VectorStoreRepository.java`의 `searchSimilar()` 메서드

### 6단계: 하이브리드 맥락 통합 및 답변 (Hybrid Generation)
확보된 **'RAG 지식'**과 **'장기 기억'**을 하나로 비벼서 **LLM**에게 최종 분석을 맡기는 단계야.
*   **설명**: 단순히 검색 결과만 던지는 게 아니라, 이전 대화의 흐름(History)과 새로 찾은 정보(RAG)를 **`RagService`**가 하나로 묶어 LLM에게 전달해. 
*   **하이브리드 효과**: "아까 그 지원자"처럼 단절된 질문도 장기 기억 덕분에 완벽하게 대답할 수 있고, 자소서라는 객관적 증거(RAG)가 더해져 답변의 신뢰도까지 최고가 되는 거지!
*   **어디에 있어?**: `RagService.java`의 `ask()` 메서드 (프롬프트 최종 조립 및 LLM 호출)

## ⚡ 스피드가 생명! 최적화 및 편의 기능

우리 시스템은 단순히 기능만 있는 게 아니라, 형의 소중한 시간을 아껴주는 디테일이 숨어있어.

### 1. 고속 부팅 (Embedding Skip)
*   **파일**: `VectorStoreRepository.java`, `RagService.java`
*   **기능**: 서버를 재기동할 때 이미 DB에 임베딩된 자소서가 있으면 작업을 **건너뛰어(Skip)**. 덕분에 1.1GB가 넘는 모델을 매번 돌릴 필요 없이 광속으로 서버가 떠!

### 2. 영구 기억 & 초기화 (Memory & Reset)
*   **파일**: `ChatController.java`, `index.html`
*   **기능**: **JPA**를 이용해 대화 내역을 DB에 영구 저장해. 페이지를 새로고침해도 대화가 이어지지! 만약 새 출발을 하고 싶다면 우측 상단의 **'초기화'** 버튼 하나면 깔끔하게 기억이 지워져.

### 3. 사용자 경험 (UX)
*   **Smooth Scroll**: 답변이 길어질 때마다 채팅창이 바닥까지 **스르륵~** 자동으로 내려가서 눈이 편해.
*   **빈 메시지 차단**: 실수로 엔터 쳐서 빈 메시지가 나가지 않도록 철저하게 막아놨어.

## 🏗️ 프로젝트 구조 및 용량 설명

우리 프로젝트 폴더들이 왜 이렇게 묵직한지, 뭐가 들어있는지 알려줄게.

### 📂 주요 폴더 구조
```text
spring-rag/
├── src/main/java/          # ☕ 자바 소스 코드 (핵심 로직)
├── src/main/resources/
│   ├── models/             # 🧠 AI 모델 파일 (1.1GB - 필수!)
│   └── static/             # 🎨 웹 화면 (HTML/Vue)
├── venv/                   # 🐍 파이썬 가상환경 (5.0GB - 준비물 도구함)
├── target/                 # 📦 빌드 결과물 (1.1GB)
└── pom.xml                 # ⚙️ 자바 라이브러리 설정 파일
```

### ⚖️ 용량 브리핑 (Total: 약 10GB+)
*   **진짜 핵심 소스 코드**: **약 200KB** (자바 파일들은 정말 가벼워!)
*   **AI의 뇌 (모델 파일)**: **1.1GB** (한국어/영어를 다 아는 백과사전이라 덩치가 커.)
*   **파이썬 도구 상자 (venv)**: **5.0GB** (AI 모델을 준비할 때 쓴 라이브러리들이야.)

## 🐍 파이썬(Python)의 역할과 은퇴

"자바 프로젝트인데 왜 파이썬이 5기가나 차지해?"라고 궁금했지? 그 이유는 바로 **'준비 단계'** 때문이야.

### 🏗️ 준비 중 (Setup Phase): 파이썬이 주인공!
*   **데이터 만들기**: 100명의 자소서를 자동으로 생성할 때 파이썬 스크립트(`generate_sample_data.py`)를 썼어.
*   **모델 다듬기**: 거대한 AI 모델을 다운로드하고, 자바가 읽을 수 있는 `ONNX` 형식으로 변환하는 작업은 파이썬 라이브러리들이 최고거든. 이때 쓴 도구들이 `venv` 폴더에 가득 찬 거야.

### 🏁 실행 중 (Runtime Phase): 파이썬은 이제 퇴근!
*   **자바의 독립**: 하지만 일단 모델 파일(`model.onnx`)이 준비되고 데이터가 DB에 들어가면, 지금 돌아가는 **스프링 부트 서버는 파이썬 없이 자바 혼자서 다 해.**
*   **은퇴 가능**: 즉, 지금 시점에서는 `venv` 폴더를 지워버려도 우리 RAG 시스템은 아주 잘 돌아가! (다만 나중에 데이터를 또 자동 생성하거나 모델을 바꿀 거라면 놔두는 게 좋지.)

## 🦾 ONNX (Open Neural Network Exchange)란?

형! 우리 프로젝트의 숨은 공신인 **ONNX**에 대해서도 간단히 설명해줄게.

### 1. 한 문장 요약
**"서로 다른 언어와 환경에서도 AI 모델을 똑같이 돌릴 수 있게 해주는 마법의 통역사"**야.

### 2. 왜 우리 프로젝트에서 써요?
*   **자바와의 만남**: 원래 대부분의 AI 모델은 파이썬(`Python`)에서 만들어져. 그런데 우리 서버는 자바(`Java`)잖아? ONNX가 있으면 파이썬에서 만든 모델을 **자바에서도 번역기 없이 직접** 돌릴 수 있어.
*   **속도가 빨라요**: ONNX는 모델을 아주 효율적으로 압축하고 실행에 최적화해줘. 그래서 CPU만으로도 형의 자소서를 빛의 속도로 숫자로 바꿀 수 있는 거야.
*   **가벼워요 (런타임 기준)**: 거대한 AI 프레임워크를 다 깔 필요 없이, `onnxruntime`이라는 작은 도구 하나만 있으면 모델을 가동할 수 있어. 

### 3. 우리 시스템에서의 역할
우리 뇌(`multilingual-e5-base`)가 **자바 서버 안에서 직접 임베딩(글자를 숫자로 바꾸는 작업)을 수행**할 수 있게 해주는 핵심 엔진이야. 이 녀석 덕분에 우리는 Ollama나 LocalAI 같은 **외부 LLM 서빙 툴에 의존하지 않고** 자바 혼자서 벡터 변환을 다 할 수 있어!

### 4. 외부 서빙 툴(Ollama 등) 대비 성능은?
| 비교 항목 | 외부 서빙 툴 (HTTP/API) | ONNX 인-프로세스 (우리 방식) |
| :--- | :--- | :--- |
| **통신 비용** | 네트워크 오버헤드 발생 (JSON/HTTP) | **제로 (자바 내부 호출)** |
| **지연 시간** | 통신 지연 존재 (수십~백 ms) | **극도로 낮음 (수 ms 이내)** |
| **복잡도** | 서빙 툴을 따로 띄우고 관리해야 함 | **자바 서버 하나면 끝!** |
| **자원 사용** | 외부 서비스가 GPU/CPU를 별도 점유 | 자바 서버의 CPU/RAM 자원 사용 |

**결론**: 우리처럼 자소서 400개를 한꺼번에 임베딩하거나, 검색할 때 짧은 문장을 벡터로 바꿀 때는 **ONNX 방식이 훨씬 빠르고 효율적**이야!

## 🤝 하이브리드 전략 (LangChain4j + Spring Ecosystem)

우리 프로젝트는 **Spring Boot 3.0.5 고정**이라는 제약 조건 속에서, 나중에 **Spring AI 1.1.2** 환경으로의 이사를 완벽하게 준비하기 위해 **'전략적 하이브리드'** 구조를 채택했어.

### 1. 각각 어디에 들어가 있나요?

*   **AI 엔진 (LangChain4j)**: `RagService.java`
    *   **역할**: 질문 분석, 벡터 검색(RAG), LLM(SiliconFlow)과의 통신 등 **AI의 핵심 두뇌** 역할을 담당해. 3.0.5 버전에서 오류 없이 가장 강력한 RAG 기능을 제공하기 때문이야.
*   **기억 및 관리 (Spring Data JPA)**: `ChatMessageRepository.java`, `ChatController.java`
    *   **역할**: 유저의 대화 내용을 DB(`test.chat_messages`)에 저장하고 불러오는 **장기 기억(Long-term Memory)** 역할을 담당해. Spring AI 1.1.2가 사용하는 데이터 구조와 유사하게 설계해서 나중에 이사 가기 딱 좋아.

### 2. 하이브리드로 쓸 때의 장단점

| 비교 항목 | 상세 내용 및 기술적 배경 |
| :--- | :--- |
| **호환성 (3.0.5 vs 3.3+)** | **Spring AI 1.1.2**는 스프링 프레임워크 6.1+에서 설계된 `RestClient`를 핵심 기반으로 사용해. 하지만 우리가 고정 중인 **Spring Boot 3.0.5**는 프레임워크 6.0 기반이라 최신 Spring AI와는 '물과 기름' 같은 사이야. 반면, **LangChain4j**는 스프링 버전과 무관하게 자바 객체(Plain Java)만으로 AI 로직을 수행할 수 있어 3.0.5에서도 최신 LLM 성능을 100% 뽑아낼 수 있는 유일한 대안이야. |
| **Advisor의 한계** | Spring AI의 **Advisor(MessageChatMemoryAdvisor 등)**는 스프링의 `Around Advice` 패턴을 이용해 LLM 호출 전후에 대화 이력을 자동으로 주입해주는 아주 편리한 '자동 필터'야. 우리 환경(3.0.5)에서는 이 자동 필터를 못 쓰기 때문에, 내가 `RagService`에서 **JPA로 대화 이력을 직접 쿼리해서 프롬프트에 수동으로 합치는 방식**을 쓴 거야. 결과적으로 기능은 동일하지만, 기술적으로는 **'스프링 AI의 자동화' 대신 'LangChain4j + JPA의 정밀 제어'**를 선택한 셈이지! |
| **커스텀 유연성** | 형이 말한 대로, **"배열만 더 갖다 붙이면 되는"** 직관적인 구조야! AOP 같은 복잡한 프레임워크 기술 없이도, 지원자의 '성향'이나 '특이사항' 같은 임의의 정보를 프롬프트 문자열 조립하듯 언제든 자유롭게 추가할 수 있어. 블랙박스 같은 자동화보다 훨씬 투명하고 강력한 제어가 가능하지! |

### 3. 왜 이렇게 했나요? (이사 전략: 무적의 진화형)
형이 나중에 이 코드를 **Spring AI 1.1.2 + Spring Boot 3.3+** 프로젝트로 가져갈 때, 단순히 이사가 끝이 아니야.

*   **진화된 미래**: 
    1.  **표준 대화 관리**: Spring AI의 `MessageChatMemoryAdvisor`를 달아서 일반적인 대화 맥락은 **자동화(AOP)**로 처리해.
    2.  **특화 정보 주입**: 동시에 우리가 만든 방식대로 **"배열만 툭 갖다 붙이는" 수동 로직**을 유지해서, 실시간 비즈니스 데이터나 특수 지침을 AI에게 아주 정밀하게 전달해.
*   **결론**: 프레임워크의 **편리함(AOP)**과 우리의 **정교한 제어(Manual)**가 결합된 **"무적의 하이브리드"** 시스템으로 완성되는 거야! 🚀

### 4. 🪟 윈도우(Windows)로 이사 가기 가이드 (Hybrid 배포)
이 프로젝트는 1.1GB가 넘는 대용량 AI 모델을 사용하므로, 안정성과 성능을 위해 **JAR 파일과 모델 폴더를 분리**하여 관리하는 것이 표준입니다.

*   **필수 보따리**:
    1.  `target/kor-rag-1.0.0.jar`: 로직과 라이브러리가 담긴 몸통 (약 150MB).
    2.  `models/` 폴더 전체: AI의 뇌에 해당하는 1.1GB 모델 데이터 (프로젝트 루트에 위치).
*   **DB 세팅**: 윈도우용 PostgreSQL을 설치하고 `test` 스키마를 생성해야 해.
*   **실행 방식**: 윈도우 터미널에서 `java -jar kor-rag-1.0.0.jar` 명령어로 실행해. 이때 반드시 `models` 폴더가 JAR 파일과 같은 위치에 있어야 AI가 모델을 찾을 수 있어! 🚀
