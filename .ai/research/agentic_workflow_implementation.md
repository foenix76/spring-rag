# Agentic Workflow 구현 방안

본 문서는 LLM 기반 애플리케이션에서 복합적인 태스크를 수행하기 위한 Agentic Workflow의 다양한 구현 전략을 기술합니다.

---

## 1. One-shot Routing (Context Injection)
**현재 프로젝트가 채택하고 있는 방식**으로, 단일 추론 단계에서 의도 파악과 필요한 데이터를 동시에 식별하여 메인 실행 에이전트에게 전달하는 구조입니다.

### 동작 원리
- **Dispatcher (Router):** 사용자의 질문을 분석하여 `INTENT`(행위)와 `QUERY`(필요 데이터 검색어)를 한 번에 추출합니다.
- **Context Loader:** 추출된 `QUERY`가 있다면 벡터 DB(RAG)를 검색하여 관련 정보를 메모리(Context)에 로드합니다.
- **Executor (Main LLM):** 로드된 정보와 함께 사용자 질문을 전달받아 최적의 도구(Tool)를 직접 호출하거나 응답을 생성합니다.

### 특징 및 장점
- **지연 시간 최소화:** 모델 호출 횟수가 최소화되어 응답 속도가 매우 빠릅니다.
- **간결한 구조:** 상태 관리나 복잡한 루프 제어가 필요 없어 시스템의 유지보수성이 높습니다.
- **적합한 사례:** 도구 한두 개로 해결 가능한 단발성 요청이나, 검색이 선행되어야 하는 직관적인 업무에 유리합니다.

---

## 2. ReAct (Reasoning + Acting)
LLM이 '생각(Thought)'과 '행동(Action)'을 번갈아 가며 수행하며, 각 단계의 '관찰(Observation)' 결과에 따라 다음 행동을 결정하는 **재귀적 루프** 방식입니다.

### 동작 원리
1. **Thought:** 질문에 대해 "현재 무엇을 해야 하는가?"를 추론합니다.
2. **Action:** 추론한 결과를 바탕으로 툴을 호출합니다.
3. **Observation:** 툴의 출력값을 확인하고 기록합니다.
4. **Repeat:** 관찰 결과를 바탕으로 다시 1번으로 돌아가거나, 최종 결론을 도출합니다.

### 살을 붙인 상세 설명
- **자가 수정(Self-correction):** 검색 결과가 예상과 다를 경우, LLM이 포기하지 않고 다른 검색어로 다시 시도하거나 계획을 즉석에서 변경할 수 있습니다.
- **도구간 데이터 연계:** 첫 번째 도구(예: 지원자 검색)에서 나온 ID를 두 번째 도구(예: 인사 기록 조회)의 입력값으로 자연스럽게 전달하는 구조입니다.
- **주의점:** 무한 루프에 빠질 위험이 있으므로 최대 반복 횟수(Max Iterations) 설정이 필수적입니다.

---

## 3. Plan-and-Execute (Planner/Worker)
작업을 시작하기 전에 상위 에이전트가 전체 실행 계획을 수립하고, 하위 에이전트들이 이를 순차적으로 실행하는 **선형적 워크플로우** 방식입니다.

### 동작 원리
- **Planner:** 전체 작업을 N개의 하위 태스크(Todo List)로 분해합니다. (예: 1. DB 조회 -> 2. CSV 생성 -> 3. 메일 발송)
- **Executor:** Planner가 작성한 리스트를 하나씩 가져와 실행만 전담하며, 각 단계의 결과를 저장합니다.
- **Re-planner:** 각 단계가 끝나면 현재 진행 상황을 보고, 남은 계획을 유지할지 수정할지 다시 판단합니다.

### 살을 붙인 상세 설명
- **복합 태스크 최적화:** 한 번의 생각으로 해결하기 어려운 거대 프로젝트를 쪼개어 처리하므로 논리적 일관성이 높습니다.
- **병렬 처리 가능:** 독립적인 태스크들이 계획에 포함되어 있다면 여러 실행기(Worker)가 동시에 작업을 처리할 수 있어 효율적입니다.
- **결과 검증:** 계획의 마지막 단계에 '검증 에이전트'를 두어 결과물이 사용자의 요구사항과 맞는지 최종 확인하는 절차를 추가하기 용이합니다.

---

## 4. Graph-based Orchestration (State Machine)
워크플로우를 상태(State)와 전이(Edge)로 정의된 그래프 구조로 설계하여, 복잡한 비즈니스 로직을 **결정론적으로 제어**하는 방식입니다.

### 동작 원리
- **Nodes:** 특정 작업(LLM 호출, 툴 실행, 외부 API 호출 등)을 수행하는 지점입니다.
- **Edges:** 조건부 전이 로직입니다. (예: 결과가 "A"이면 Node-2로, "B"이면 Node-3으로 이동)
- **State:** 그래프 전체에서 공유되는 상태 객체로, 각 노드에서의 결과물이 여기에 누적 저장됩니다.

### 살을 붙인 상세 설명
- **엔터프라이즈급 안정성:** LLM의 자유도에만 의존하지 않고, 인간이 설계한 비즈니스 룰을 강제할 수 있습니다. (예: "메일 발송 전에는 반드시 승인 노드를 거쳐야 함")
- **체크포인트 및 복구:** 상태가 그래프 단위로 관리되므로, 중간에 실패하더라도 실패한 지점부터 다시 시작하는 '재시도 로직' 구현이 매우 강력합니다.
- **순환 구조 지원:** 작업이 완료될 때까지 특정 구간을 반복(Loop)하거나, 특정 조건 만족 시에만 탈출하는 정교한 제어가 가능합니다. (LangGraph가 이 방식의 대표 주자입니다.)
