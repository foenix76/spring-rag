# 🛠️ 필드 원천 분리형 스트리밍 및 RAG 로직 전면 재건축 계획

## 1. 개요 (Goal)
Gemini가 초래한 "문자열 파싱 기반 스트리밍"의 취약성을 해결하고, 형(유저)이 설계한 **"필드 원천 분리형 멀티 채널 스트리밍"** 아키텍처를 완벽하게 구현합니다. LLM이 툴 결과를 읊는 것을 기다리지 않고, 백엔드에서 툴 실행 즉시 이벤트를 가로채서 프론트엔드로 쏴주는 견고한 구조를 만드는 것이 핵심입니다.

## 2. 사용자 검토 필요 사항
> [!IMPORTANT]
> **필드 분리 프로토콜 정의**: 이제 SSE 스트림은 단순 텍스트가 아닌 JSON 객체(`{text: "...", system: "...", approval: "...", completed: "..."}`)를 고정적으로 반환합니다. 프론트엔드는 이 구조를 100% 신뢰해야 합니다.
> 
> **Interceptor 방식 선택**: Spring AI의 `ChatClient.Advisor`를 사용하여 툴 호출 및 응답 스트림을 가로챕니다. 이는 `RagService` 코드를 훨씬 깔끔하게 유지해 줄 것입니다.

## 4. 상세 구현 전략 (Surgical Interceptor)

### A. 백엔드: Advisor를 통한 툴 결과 가로채기 컨셉 (Few-shot)
Spring AI의 `CallAroundAdvisor`를 활용하여 툴이 뱉는 `[APPROVAL:...]` 토큰을 LLM에게 가기 전에 가로앱니다.

```java
// Interceptor 예시
public class MultiChannelStreamAdvisor implements CallAroundAdvisor {
    @Override
    public ChatResponse aroundCall(AnyRequest request, CallAroundAdvisorChain chain) {
        ChatResponse response = chain.nextAroundCall(request);
        // 툴 실행 결과(Assistant Message)를 뒤져서 특정 토큰이 있으면
        // 응답 메타데이터에 'structural_event'로 실어서 반환
        return response;
    }
}
```

### B. RAG 검색어 추출 Few-shot (RAG Precision)
LLM이 질문에서 검색어를 더 정확하게 뽑도록 System Prompt를 강화합니다.

```markdown
[Few-shot]
사용자: "올해 합격한 지원자들 리스트 보여줘"
AI: INTENT: [SEARCH] QUERY: "합격여부:PASS"
사용자: "이 사람(ID123)한테 불합격 메일 보내"
AI: INTENT: [ACTION] QUERY: "ID:123"
```

## 5. 확인 질문
1. **히스토리 저장 방식**: 현재 `ChatMessage` 엔티티는 단일 `content` 필드만 가집니다. 필드 분리 정보를 완벽히 복구하려면 JSON으로 통째로 저장하거나, content에 구분자를 두고 저장해야 합니다. 어떤 방식을 선호하시나요? (기존 방식 유지 시 강한 정규식 복원이 필요합니다) -> 기존데이터 다 날릴거야. json으로 통저장하자.
2. **이동 도구(navigateTo)**: 이 도구는 `NavigationResult` 레코드를 반환하는데, 현재 UI에서 '시스템 로그'로 처리되고 있습니다. 별도의 하단 알림창(Toast)으로 띄울까요, 아니면 지금처럼 대화 흐름에 둘까요? -> POC니까 연출효과를 위해 지금처럼 시스템 로그는 출력해주고 별도의 하단 알림창(Toast)은 떴다 2초후 사라지게만 해줘. 너무 공들일 필요는 없다.
